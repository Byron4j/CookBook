# 负载均衡算法

## 1. 概念

负载均衡（Load Balancing）是将请求分发到多个后端服务器的技术，提高系统整体处理能力和可用性。

```
负载均衡架构:

        用户请求
           │
           ▼
    ┌─────────────┐
    │   DNS 层     │ ← 全局负载均衡（GSLB）
    └──────┬──────┘
           │
           ▼
    ┌─────────────┐
    │  LVS/Nginx  │ ← 四层/七层负载均衡
    └──────┬──────┘
           │
      ┌────┼────┐
      ▼    ▼    ▼
   [S1] [S2] [S3]  ← 后端服务器集群
```

## 2. 来源

负载均衡技术源于互联网高速发展期的扩容需求。  
从硬件负载均衡器（F5、A10）到软件方案（Nginx、LVS、HAProxy），再到云原生方案（Kubernetes Service、Service Mesh）。

## 3. 知识点

### 3.1 轮询（Round Robin）

```
轮询算法:

请求序列: R1, R2, R3, R4, R5, R6, R7

服务器: [S1, S2, S3]

R1 → S1
R2 → S2
R3 → S3
R4 → S1
R5 → S2
R6 → S3
R7 → S1

特点:
- 实现最简单
- 请求均匀分布
- 不考虑服务器性能差异
- 不考虑服务器当前负载

加权轮询:
服务器性能不同，分配不同权重:

S1: weight=3
S2: weight=2
S3: weight=1

序列: S1, S1, S1, S2, S2, S3, S1, S1, S1, S2, S2, S3, ...

平滑加权轮询（Nginx 算法）:
解决普通加权轮询的突发问题

初始: current_weight = [0, 0, 0]
      effective_weight = [3, 2, 1]
      total_weight = 6

R1: current += effective → [3, 2, 1]
    选最大的 S1 (3)
    S1 的 current -= total → [3-6, 2, 1] = [-3, 2, 1]
    
R2: current += effective → [0, 4, 2]
    选最大的 S2 (4)
    S2 的 current -= total → [0, 4-6, 2] = [0, -2, 2]
    
R3: current += effective → [3, 0, 3]
    选 S1 (3)
    S1 的 current -= total → [3-6, 0, 3] = [-3, 0, 3]
    
R4: current += effective → [0, 2, 4]
    选 S3 (4)
    S3 的 current -= total → [0, 2, 4-6] = [0, 2, -2]

结果: S1, S2, S1, S3, S1, S2, S1, S3, ...
分布更平滑！
```

**Java 实现:**
```java
public class RoundRobinLoadBalancer {
    private final List<String> servers;
    private final AtomicInteger currentIndex = new AtomicInteger(0);
    
    public RoundRobinLoadBalancer(List<String> servers) {
        this.servers = servers;
    }
    
    public String select() {
        if (servers.isEmpty()) return null;
        int index = currentIndex.getAndIncrement() % servers.size();
        return servers.get(index);
    }
}

// 加权轮询
public class WeightedRoundRobin {
    private final List<Server> servers;
    private final AtomicInteger currentIndex = new AtomicInteger(0);
    
    static class Server {
        String address;
        int weight;
        int currentWeight = 0;
        
        Server(String address, int weight) {
            this.address = address;
            this.weight = weight;
        }
    }
    
    public synchronized String select() {
        int totalWeight = 0;
        Server maxServer = null;
        
        for (Server server : servers) {
            totalWeight += server.weight;
            server.currentWeight += server.weight;
            
            if (maxServer == null || server.currentWeight > maxServer.currentWeight) {
                maxServer = server;
            }
        }
        
        if (maxServer != null) {
            maxServer.currentWeight -= totalWeight;
            return maxServer.address;
        }
        
        return null;
    }
}
```

### 3.2 随机（Random）

```
随机算法:

每次请求随机选择一个服务器:

R1 → random() = 0.3 → S1
R2 → random() = 0.8 → S3
R3 → random() = 0.1 → S1
R4 → random() = 0.5 → S2
R5 → random() = 0.9 → S3

特点:
- 实现简单
- 长时间看分布均匀
- 短时间可能不均
- 适合无状态服务

加权随机:
根据权重比例选择:

S1: weight=5 (50%)
S2: weight=3 (30%)
S3: weight=2 (20%)

生成 0-9 的随机数:
0-4 → S1
5-7 → S2
8-9 → S3
```

### 3.3 最少连接（Least Connections）

```
最少连接算法:

维护每个服务器的当前连接数:

S1: connections=10
S2: connections=5  ← 最少，选择 S2
S3: connections=8

R1 → S2 (connections: 5→6)

R2:
S1: 10
S2: 6
S3: 8  ← 最少，选择 S3

R2 → S3 (connections: 8→9)

特点:
- 考虑服务器实际负载
- 适合长连接场景（WebSocket）
- 需要维护连接计数器
- 短期可能不均

加权最少连接:
S1: weight=5, connections=10 → ratio=2.0
S2: weight=3, connections=3  → ratio=1.0 ← 选择 S2
S3: weight=2, connections=8  → ratio=4.0
```

### 3.4 源地址哈希（Source IP Hash）

```
源地址哈希:

根据客户端 IP 计算哈希值，选择服务器:

server = hash(clientIP) % serverCount

Client A (192.168.1.1):
hash = 12345 % 3 = 0 → S1

Client B (192.168.1.2):
hash = 67890 % 3 = 2 → S3

Client C (192.168.1.3):
hash = 54321 % 3 = 0 → S1

特点:
- 同一客户端始终访问同一服务器
- 天然会话保持（Session Affinity）
- 服务器增减时影响较大
- 可能导致数据倾斜

一致性哈希改进:
使用一致性哈希环
减少服务器增减时的影响
```

### 3.5 健康检查

```
健康检查机制:

        负载均衡器
            │
      ┌─────┼─────┐
      ▼     ▼     ▼
    [S1]  [S2]  [S3]
      ↑     ×     ↑
     健康   故障   健康
      
S2 被标记为不健康，不再接收请求

健康检查方式:

1. 主动检查（Active）
   - 负载均衡器定期 ping 服务器
   - TCP 连接探测
   - HTTP 健康检查接口
   
   示例:
   GET /health
   返回 200 → 健康
   返回 5xx 或无响应 → 故障

2. 被动检查（Passive）
   - 监控实际请求的响应
   - 连续 3 次失败 → 标记故障
   - 连续 3 次成功 → 恢复服务

3. 组合检查
   - 主动 + 被动
   - 更准确，减少误判

故障恢复:
- 摘除故障节点
- 请求转发到健康节点
- 故障节点恢复后重新加入
```

## 4. 生产实践

### 4.1 Nginx 负载均衡配置

```
Nginx 负载均衡配置示例:

http {
    upstream backend {
        # 轮询（默认）
        server 192.168.1.1:8080 weight=3;
        server 192.168.1.2:8080 weight=2;
        server 192.168.1.3:8080 backup;  # 备用
        
        # 健康检查（需要 nginx_upstream_check_module）
        check interval=3000 rise=2 fall=3 timeout=1000 type=http;
        check_http_send "GET /health HTTP/1.0\r\n\r\n";
        check_http_expect_alive http_2xx http_3xx;
    }
    
    upstream websocket_backend {
        # 最少连接（适合 WebSocket）
        least_conn;
        
        server 192.168.1.1:8080;
        server 192.168.1.2:8080;
        server 192.168.1.3:8080;
    }
    
    upstream session_backend {
        # IP 哈希（会话保持）
        ip_hash;
        
        server 192.168.1.1:8080;
        server 192.168.1.2:8080;
    }
    
    server {
        listen 80;
        
        location / {
            proxy_pass http://backend;
        }
        
        location /ws {
            proxy_pass http://websocket_backend;
            proxy_http_version 1.1;
            proxy_set_header Upgrade $http_upgrade;
            proxy_set_header Connection "upgrade";
        }
    }
}
```

自定义负载均衡规则:

public class CustomRule extends AbstractLoadBalancerRule {
    @Override
    public Server choose(Object key) {
        List<Server> servers = getLoadBalancer().getReachableServers();
        
        if (servers.isEmpty()) return null;
        
        // 自定义选择逻辑
        // 例如: 选择 CPU 负载最低的服务器
        return servers.stream()
            .min(Comparator.comparingInt(this::getCpuLoad))
            .orElse(servers.get(0));
    }
    
    private int getCpuLoad(Server server) {
        // 从监控中心获取 CPU 负载
        return MonitoringService.getCpuLoad(server.getHost());
    }
}

## 5. 最佳实践

1. **无状态服务用轮询或随机**，简单高效。  
2. **长连接场景用最少连接**，避免单点过载。  
3. **需要会话保持用源地址哈希或一致性哈希**。  
4. **必须配置健康检查**，及时摘除故障节点。  
5. **监控每个后端服务器的负载**，动态调整权重。  
6. **设置合理的超时和重试机制**。  
7. **考虑地域和可用区**，优先本地调用。  

### 负载均衡算法选择

```
决策树:

需要会话保持？
  ├── 是 → 源地址哈希 / 一致性哈希
  │         - 用户始终访问同一服务器
  │         - 适合有状态服务
  │
  └── 否 → 连接是否长连接？
             ├── 是 → 最少连接
             │         - WebSocket
             │         - 数据库连接池
             │
             └── 否 → 服务器性能是否相同？
                        ├── 是 → 轮询 / 随机
                        │         - 最简单
                        │         - 最常用
                        │
                        └── 否 → 加权轮询 / 加权随机
                                  - 根据性能分配权重
                                  - 高性能多分配

生产环境推荐:
- 网关层: 轮询 + 健康检查
- 微服务: Ribbon 随机 + 重试
- 数据库: 最少连接
- 缓存: 一致性哈希
```
