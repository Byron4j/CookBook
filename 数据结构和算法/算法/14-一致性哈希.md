# 一致性哈希

## 1. 概念

一致性哈希（Consistent Hashing）是一种特殊的哈希算法，在分布式系统中用于解决节点动态增减时的数据迁移问题。

```
传统哈希 vs 一致性哈希:

传统哈希:
server = hash(key) % N  (N = 服务器数量)

问题: 当 N 变化时，几乎所有 key 都需要重新映射！

N=3: hash(key) % 3
key1 → server 0
key2 → server 1
key3 → server 2

N=4: hash(key) % 4
key1 → server 1 (变了！)
key2 → server 2 (变了！)
key3 → server 3 (变了！)
数据迁移率: 75% (3/4)


一致性哈希:
将节点和 key 映射到一个环上

       hash环 (0 ~ 2^32-1)
       
       0° ──────────────────→ 360°
            
            [Node A]  hash=50
               
            [Key 1]   hash=60  → 顺时针找到 Node A
            
            [Key 2]   hash=150 → 顺时针找到 Node B
            
            [Node B]  hash=200
            
            [Key 3]   hash=250 → 顺时针找到 Node B
            
            [Node C]  hash=350
            
            [Key 4]   hash=400 → 顺时针找到 Node C (绕回)
            
            [Key 5]   hash=30  → 顺时针找到 Node A (绕回)

添加 Node D (hash=100):
- 只影响 Key 1 (hash=60)，从 Node A 迁移到 Node D
- Key 2,3,4,5 不受影响
数据迁移率: 20% (1/5)
```

## 2. 来源

一致性哈希由 David Karger 等人于 1997 年提出。  
广泛应用于分布式缓存（Memcached、Redis Cluster）、负载均衡、分布式存储等场景。

## 3. 知识点

### 3.1 基础一致性哈希

```
哈希环结构:

       0 (0°)
       │
   270°├────────┐
       │        │
       │   环    │
       │        │
       └────────┘90°
       
       180°

节点映射:
Node A: hash("NodeA") = 50
Node B: hash("NodeB") = 200
Node C: hash("NodeC") = 350

Key 映射规则:
顺时针找到第一个节点

Key 1: hash("key1") = 60  → Node A (50 < 60 ≤ 200? 不对)
       
       应该是: 顺时针从 60 开始找
       60 → 200 (Node B)
       
       Wait, 让我重新理解:
       
       0° ──────────────────→ 360°
       
       Node A @ 50
       Node B @ 200
       Node C @ 350
       
       Key 1 @ 60:
       顺时针: 60 → 200 (Node B)
       
       Key 2 @ 150:
       顺时针: 150 → 200 (Node B)
       
       Key 3 @ 250:
       顺时针: 250 → 350 (Node C)
       
       Key 4 @ 400 (绕回):
       顺时针: 400 → 0 → 50 (Node A)
       
       Key 5 @ 30:
       顺时针: 30 → 50 (Node A)

节点增减影响:

添加 Node D @ 100:
       
       原来归属:
       Key 1 @ 60 → Node B
       
       现在:
       Key 1 @ 60 → Node D (60 < 100)
       
       只影响 50~100 区间的 Key！
       其他 Key 不受影响

删除 Node B:
       
       原来归属 Node B 的 Key:
       Key 1 @ 60, Key 2 @ 150
       
       现在:
       Key 1 @ 60 → Node C? No
       Key 1 @ 60 → 顺时针: 60 → 100 (Node D, 如果存在)
       
       如果没有 Node D:
       Key 1 @ 60 → 顺时针: 60 → 350 (Node C) 绕回
       Key 2 @ 150 → 顺时针: 150 → 350 (Node C) 绕回
       
       只影响原来 Node B 负责的 Key
```

### 3.2 虚拟节点

```
虚拟节点解决数据倾斜:

问题:
如果节点数量少，可能导致数据分布不均

Node A @ 50
Node B @ 200
Node C @ 350

Key 分布可能严重不均:
0-50:    Node A (50)
50-200:  Node B (150)  ← 负责最多
200-350: Node C (150)
350-0:   Node A (60)


解决方案: 虚拟节点
每个物理节点对应多个虚拟节点

Node A: A#1, A#2, A#3, A#4
Node B: B#1, B#2, B#3, B#4
Node C: C#1, C#2, C#3, C#4

hash("A#1") = 30
hash("A#2") = 90
hash("A#3") = 180
hash("A#4") = 330
hash("B#1") = 50
hash("B#2") = 120
hash("B#3") = 240
hash("B#4") = 310
hash("C#1") = 10
hash("C#2") = 150
hash("C#3") = 260
hash("C#4") = 340

现在环上更均匀:
0-10:   C#1 (Node C)
10-30:  A#1 (Node A)
30-50:  C#1 (Node C) → No, A#1=30, B#1=50
        
        让我重新整理环上顺序:
        
        10: C#1 (Node C)
        30: A#1 (Node A)
        50: B#1 (Node B)
        90: A#2 (Node A)
        120: B#2 (Node B)
        150: C#2 (Node C)
        180: A#3 (Node A)
        240: B#3 (Node B)
        260: C#3 (Node C)
        310: B#4 (Node B)
        330: A#4 (Node A)
        340: C#4 (Node C)
        
        每个物理节点负责的数据更均匀！
        
        添加/删除节点时:
        每个物理节点有多个虚拟节点
        影响被分散到多个区间
        数据迁移更均匀
```

### 3.3 时间复杂度

```
查找时间复杂度:

线性查找: O(N)  (N = 节点数)
- 遍历所有节点，找到顺时针第一个

二分查找: O(log N)
- 节点按哈希值排序
- 用 TreeMap 或有序数组

最佳实践:
使用 TreeMap（红黑树）存储节点
查找: TreeMap.ceilingEntry(hash) → O(log N)

Java 实现:
TreeMap<Integer, Node> ring = new TreeMap<>();
ring.put(hash("NodeA"), nodeA);
ring.put(hash("NodeB"), nodeB);

// 查找
Map.Entry<Integer, Node> entry = ring.ceilingEntry(hash("key1"));
if (entry == null) {
    entry = ring.firstEntry();  // 绕回
}
Node target = entry.getValue();
```

## 4. 数据结构

```java
import java.security.MessageDigest;
import java.security.NoSuchAlgorithmException;
import java.util.*;

public class ConsistentHashRouter {
    
    private final TreeMap<Long, String> ring = new TreeMap<>();
    private final int virtualNodeCount;  // 每个物理节点的虚拟节点数
    
    public ConsistentHashRouter(int virtualNodeCount) {
        this.virtualNodeCount = virtualNodeCount;
    }
    
    /**
     * 添加节点
     */
    public void addNode(String node) {
        for (int i = 0; i < virtualNodeCount; i++) {
            String virtualNode = node + "#" + i;
            long hash = hash(virtualNode);
            ring.put(hash, node);
        }
    }
    
    /**
     * 删除节点
     */
    public void removeNode(String node) {
        for (int i = 0; i < virtualNodeCount; i++) {
            String virtualNode = node + "#" + i;
            long hash = hash(virtualNode);
            ring.remove(hash);
        }
    }
    
    /**
     * 根据 key 获取节点
     */
    public String getNode(String key) {
        if (ring.isEmpty()) {
            return null;
        }
        
        long hash = hash(key);
        
        // 找到顺时针方向的第一个节点
        Map.Entry<Long, String> entry = ring.ceilingEntry(hash);
        if (entry == null) {
            // 绕回环的头部
            entry = ring.firstEntry();
        }
        
        return entry.getValue();
    }
    
    /**
     * 获取所有节点（去重）
     */
    public Set<String> getAllNodes() {
        return new HashSet<>(ring.values());
    }
    
    /**
     * MurmurHash 算法（高效、低碰撞）
     */
    private long hash(String key) {
        try {
            MessageDigest md = MessageDigest.getInstance("MD5");
            byte[] digest = md.digest(key.getBytes());
            
            // 取前 8 字节作为 long
            long hash = 0;
            for (int i = 0; i < 8; i++) {
                hash <<= 8;
                hash |= (digest[i] & 0xFF);
            }
            return hash;
        } catch (NoSuchAlgorithmException e) {
            // 回退到简单哈希
            return key.hashCode() & 0xFFFFFFFFL;
        }
    }
    
    /**
     * 打印环的状态
     */
    public void printRing() {
        System.out.println("一致性哈希环状态:");
        for (Map.Entry<Long, String> entry : ring.entrySet()) {
            System.out.printf("Hash: %d → Node: %s%n", 
                entry.getKey(), entry.getValue());
        }
    }
}

// 使用示例
public class ConsistentHashExample {
    public static void main(String[] args) {
        ConsistentHashRouter router = new ConsistentHashRouter(150);  // 150 个虚拟节点
        
        // 添加节点
        router.addNode("Server1");
        router.addNode("Server2");
        router.addNode("Server3");
        
        // 查询 key 归属
        String[] keys = {"user:1", "user:2", "user:3", "order:100", "product:50"};
        for (String key : keys) {
            System.out.println(key + " → " + router.getNode(key));
        }
        
        // 添加新节点
        System.out.println("\n添加 Server4 后:");
        router.addNode("Server4");
        
        for (String key : keys) {
            System.out.println(key + " → " + router.getNode(key));
        }
    }
}
```

## 5. 生产实践

### 5.1 Redis Cluster 分片

```
Redis Cluster 使用一致性哈希:

数据分片:
Key → CRC16(key) % 16384 → Slot

16384 个 slot 分配到各个节点:
Node A: 0-5460
Node B: 5461-10922
Node C: 10923-16383

添加节点 D:
从现有节点迁移部分 slot 到 D
只影响被迁移 slot 的数据
其他 slot 完全不受影响

槽位迁移过程:
1. 新节点加入集群
2. 分配 slot 范围（如 0-1365）
3. 源节点将这些 slot 的数据迁移到新节点
4. 迁移过程中，请求可能被重定向（MOVED/ASK）

优势:
- 数据迁移量可控
- 大部分请求不受影响
- 集群可以在线扩容
```

### 5.2 负载均衡

```
一致性哈希负载均衡:

用户请求 → 负载均衡器 → 后端服务器

传统轮询:
Request 1 → Server 1
Request 2 → Server 2
Request 3 → Server 3
Request 4 → Server 1

问题: 同一个用户的请求可能打到不同服务器
      （Session 共享问题）

一致性哈希负载均衡:
Request 1 (user=A) → hash("user=A") → Server 2
Request 2 (user=A) → hash("user=A") → Server 2  (相同！)
Request 3 (user=B) → hash("user=B") → Server 1

优势:
- 同一用户始终路由到同一服务器
- 服务器增减时影响最小
- 天然支持会话保持

Nginx 配置:
http {
    upstream backend {
        consistent_hash $request_uri;  # 基于 URI 一致性哈希
        server 192.168.1.1:8080;
        server 192.168.1.2:8080;
        server 192.168.1.3:8080;
    }
}
```

### 5.3 分布式缓存

```
Memcached 一致性哈希:

缓存分片:

Client 1 ──┐
Client 2 ──┼──→ 一致性哈希环 ──→ [Cache A] [Cache B] [Cache C]
Client 3 ──┘

添加 Cache D:
- 只影响部分 key
- 大部分缓存仍然有效
- 避免缓存雪崩

对比普通哈希:

普通哈希: 添加节点 → 90%+ 缓存失效 → 数据库压力激增
一致性哈希: 添加节点 → 少量缓存失效 → 平滑过渡
```

### 5.4 数据倾斜处理

```
数据倾斜问题:

某些热点 key 集中在一个节点:

Node A: [user:1, user:2, user:3, ..., user:10000]  ← 10k 用户
Node B: [user:10001, user:10002]                     ← 2 个用户
Node C: []                                           ← 空

解决方案:

1. 增加虚拟节点数
   - 每个物理节点 150~200 个虚拟节点
   - 数据分布更均匀

2. 局部改造
   - 对热点 key 添加随机后缀
   - user:1 → user:1_0, user:1_1, user:1_2
   - 分散到多个节点

3. 自定义哈希函数
   - 使用 MurmurHash 替代普通哈希
   - 分布更均匀，碰撞更少

4. 监控和自动均衡
   - 监控每个节点的负载
   - 自动迁移数据
```

## 最佳实践

1. **虚拟节点数建议 150~200**，平衡性能和分布均匀性。  
2. **使用 MurmurHash 或 MD5**，避免简单哈希的碰撞。  
3. **节点增减时做数据预热**，避免缓存击穿。  
4. **监控数据分布**，及时发现热点和倾斜。  
5. **结合副本策略**，提高可用性。  
6. **考虑节点的权重差异**（异构硬件）。
