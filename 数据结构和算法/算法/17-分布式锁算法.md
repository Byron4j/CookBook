# 分布式锁算法

## 1. 概念

分布式锁是在分布式系统中控制多个进程/节点对共享资源互斥访问的机制。

```
分布式锁场景:

库存扣减:
用户 A ──→ 查询库存: 100
用户 B ──→ 查询库存: 100
              │
              ▼
用户 A ──→ 扣减库存: 99
用户 B ──→ 扣减库存: 99  (应该是 98！)

问题: 并发导致超卖

解决方案: 分布式锁

用户 A ──→ 获取锁 ──→ 查询: 100 ──→ 扣减: 99 ──→ 释放锁
              ↑                              │
              └──────────────────────────────┘
              
用户 B ──→ 获取锁（等待）
              │
              ▼
          用户 A 释放后
              │
              ▼
          获取成功
              │
              ▼
          查询: 99 ──→ 扣减: 98 ──→ 释放锁

三种实现方案:
- 基于数据库
- 基于 Redis
- 基于 Zookeeper / etcd
```

## 2. 来源

分布式锁是分布式系统的基石算法之一。  
随着微服务和分布式架构的普及，分布式锁成为必备组件。

## 3. 知识点

### 3.1 基于数据库

```
数据库实现方式:

方案 1: 唯一索引

CREATE TABLE distributed_lock (
    lock_name VARCHAR(64) NOT NULL PRIMARY KEY,
    locked_by VARCHAR(64),
    expire_time TIMESTAMP
);

加锁:
INSERT INTO distributed_lock (lock_name, locked_by, expire_time)
VALUES ('order_lock', 'server1', DATE_ADD(NOW(), INTERVAL 30 SECOND));

- 成功: 获得锁
- 失败（唯一键冲突）: 锁被占用

解锁:
DELETE FROM distributed_lock WHERE lock_name = 'order_lock';

问题:
- 性能差（数据库操作慢）
- 单点故障
- 无法自动续期


方案 2: FOR UPDATE

BEGIN;
SELECT * FROM distributed_lock 
WHERE lock_name = 'order_lock' 
FOR UPDATE;

- 获得行锁
- 其他事务阻塞

COMMIT;  -- 释放锁

问题:
- 事务粒度难控制
- 性能差
- 可能死锁
```

### 3.2 基于 Redis

```
Redis 实现方式:

方案 1: SETNX + EXPIRE

SETNX lock:order true    -- 不存在才设置
EXPIRE lock:order 30     -- 30秒过期

问题: SETNX 和 EXPIRE 非原子操作
      如果 SETNX 后崩溃，锁永不释放

方案 2: SET key value NX EX seconds（Redis 2.6.12+）

SET lock:order random_value NX EX 30

- NX: 不存在才设置
- EX: 设置过期时间（秒）
- PX: 设置过期时间（毫秒）

random_value: 唯一标识（UUID），防止误删

解锁:
if (redis.get("lock:order") == random_value) {
    redis.del("lock:order");
}

问题: GET + DEL 非原子操作
      可能判断通过后被其他线程获取


方案 3: Lua 脚本（原子操作）

加锁:
```lua
-- 尝试获取锁
if redis.call("setnx", KEYS[1], ARGV[1]) == 1 then
    redis.call("expire", KEYS[1], ARGV[2])
    return 1
end
return 0
```

解锁:
```lua
-- 安全释放锁
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

方案 4: RedLock（Redis 作者提出）

在多个独立 Redis 节点上加锁:
- 向 5 个节点发送加锁请求
- 如果超过半数（3个）成功，且耗时小于锁过期时间，则认为加锁成功
- 解锁时向所有节点发送解锁请求

优点:
- 容忍部分节点故障

缺点:
- 实现复杂
- 性能较差
- 时钟漂移问题

争议:
Martin Kleppmann 认为 RedLock 不可靠
建议使用 fencing token 保证安全
```

**Java 实现（Redisson）:**
```java
// Redisson 分布式锁
@Configuration
public class RedissonConfig {
    @Bean
    public RedissonClient redissonClient() {
        Config config = new Config();
        config.useSingleServer()
            .setAddress("redis://localhost:6379")
            .setPassword("password");
        return Redisson.create(config);
    }
}

@Service
public class OrderService {
    @Autowired
    private RedissonClient redissonClient;
    
    public void deductStock(Long productId) {
        String lockKey = "lock:stock:" + productId;
        RLock lock = redissonClient.getLock(lockKey);
        
        try {
            // 尝试获取锁，等待10秒，锁30秒自动释放
            boolean locked = lock.tryLock(10, 30, TimeUnit.SECONDS);
            if (locked) {
                // 扣减库存
                stockDao.deduct(productId);
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        } finally {
            // 释放锁
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

### 3.3 基于 Zookeeper

```
Zookeeper 实现方式:

Zookeeper 特性:
- 临时顺序节点（EPHEMERAL_SEQUENTIAL）
- 节点删除时自动通知 Watcher
- 天然支持等待队列

加锁流程:

1. 在 /locks 下创建临时顺序节点:
   /locks/lock-00000001
   
2. 获取 /locks 下所有子节点，排序

3. 如果当前节点是最小序号，获得锁

4. 否则，监听前一个节点的删除事件
   当前: /locks/lock-00000003
   监听: /locks/lock-00000002
   
5. 前一个节点删除后，当前节点获得锁

解锁流程:

1. 删除当前临时节点
2. 客户端断开连接，节点自动删除

优点:
- 天然防死锁（客户端断开自动释放）
- 支持等待队列（公平锁）
- 无需设置过期时间

缺点:
- 性能不如 Redis
- 依赖 Zookeeper 集群
- 网络抖动可能导致误释放


Curator 框架实现:

InterProcessMutex lock = new InterProcessMutex(client, "/locks/order");
try {
    if (lock.acquire(10, TimeUnit.SECONDS)) {
        // 执行业务逻辑
    }
} finally {
    lock.release();
}
```

## 4. 生产实践

### 4.1 锁的续期问题

```
锁续期问题:

业务执行时间 > 锁过期时间:

T0: 获取锁，过期时间 30s
T10: 业务还在执行
T20: 业务还在执行
T30: 锁自动过期！其他线程可以获取锁
T35: 当前线程执行完，释放锁（可能是别人的锁）

解决方案:

1. Watch Dog（看门狗）
   Redisson 实现:
   - 获取锁后，启动定时任务
   - 每 10 秒检查锁是否还持有
   - 如果持有，续期到 30 秒
   - 业务完成，停止定时任务
   
2. 业务时间预估
   - 设置足够长的过期时间
   - 缺点: 崩溃时恢复时间长

3. 版本号机制
   - 锁带有版本号
   - 释放时检查版本号
   - 防止误删
```

### 4.2 方案对比

```
分布式锁方案对比:

            数据库      Redis      Zookeeper    etcd
─────────────────────────────────────────────────────
性能:        低         高          中           高
可靠性:      中         高          高           高
实现复杂度:   低         中          中           中
死锁处理:     差         好          优秀         优秀
等待队列:     无         无          有           有
可重入:       无         有          有           有
自动续期:     无         有          不需要       不需要

推荐:
- 一般场景: Redis + Redisson
- 强一致性: Zookeeper / etcd
- 简单场景: 数据库（不推荐生产环境）
```

### 4.3 实际应用

```
电商库存扣减:

@Service
public class StockService {
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    private static final String STOCK_KEY = "stock:%d";
    private static final String LOCK_KEY = "lock:stock:%d";
    
    public boolean deductStock(Long productId, int quantity) {
        String lockKey = String.format(LOCK_KEY, productId);
        String stockKey = String.format(STOCK_KEY, productId);
        String requestId = UUID.randomUUID().toString();
        
        // 1. 获取锁
        Boolean locked = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, requestId, 30, TimeUnit.SECONDS);
        
        if (!Boolean.TRUE.equals(locked)) {
            return false;  // 获取锁失败
        }
        
        try {
            // 2. 检查库存
            String stockStr = redisTemplate.opsForValue().get(stockKey);
            int stock = Integer.parseInt(stockStr);
            
            if (stock < quantity) {
                return false;  // 库存不足
            }
            
            // 3. 扣减库存
            redisTemplate.opsForValue().decrement(stockKey, quantity);
            
            // 4. 异步同步数据库
            asyncUpdateDatabase(productId, quantity);
            
            return true;
        } finally {
            // 5. 释放锁（Lua 脚本保证原子性）
            String script = 
                "if redis.call('get', KEYS[1]) == ARGV[1] then " +
                "return redis.call('del', KEYS[1]) " +
                "else return 0 end";
            
            redisTemplate.execute(
                new DefaultRedisScript<>(script, Long.class),
                Collections.singletonList(lockKey),
                requestId
            );
        }
    }
}
```

## 5. 最佳实践

1. **优先使用成熟的框架**（Redisson、Curator）。  
2. **锁的粒度尽量小**（细化到具体资源）。  
3. **锁的过期时间要合理**（业务时间 + 缓冲）。  
4. **必须处理锁续期**（防止业务未完成锁失效）。  
5. **解锁时要判断是否是当前线程的锁**。  
6. **使用 UUID 作为锁的值**，防止误删。  
7. **考虑锁的可重入性**。  
8. **做好降级方案**（获取锁失败时的处理）。  

### 分布式锁的陷阱

```
常见陷阱:

1. 锁过期时间太短
   - 业务未完成，锁已释放
   - 其他线程获取锁，数据混乱
   
2. 不解锁或重复解锁
   - 死锁
   - 误删其他线程的锁
   
3. 锁范围过大
   - 并发度降低
   - 性能下降
   
4. 忽略网络延迟
   - 锁在传输过程中过期
   - 脑裂问题

5. 单点故障
   - Redis 主从切换时数据丢失
   - 使用 RedLock 或多主架构
```
