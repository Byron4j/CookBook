# B树

## 1. 概念

B 树是多路平衡查找树，常用于磁盘/SSD 等外存索引场景。  
一个节点可包含多个 key 和多个子节点，树高通常很低。

```
3 阶 B 树（2-3 树）示例:

            [30]
           /    \
        [20]    [40, 50]
        /  \    /   |   \
     [10] [25][35] [45] [60, 70]

特点:
- 每个节点可以有多个 key
- 每个节点可以有多个子节点（子节点数 = key 数 + 1）
- 所有叶子节点在同一层
- 节点内 key 有序
```

## 2. 来源

B 树诞生于数据库和文件系统索引需求：  
相较二叉树，B 树通过"多路分支 + 高扇出"减少磁盘 I/O 次数。

## 3. 知识点

### 3.1 B 树的阶（Order）

```
m 阶 B 树的性质:

1. 每个节点最多有 m 个子节点
2. 每个节点最多有 m-1 个 key
3. 根节点最少有 2 个子节点（如果不是叶子）
4. 非根非叶子节点最少有 ⌈m/2⌉ 个子节点
5. 所有叶子节点在同一层

示例对比:

2 阶 B 树 = 二叉搜索树
3 阶 B 树（2-3 树）: 每个节点 1~2 个 key，2~3 个子节点
4 阶 B 树（2-3-4 树）: 每个节点 1~3 个 key，2~4 个子节点

实际应用:
- MySQL InnoDB: 16KB 页大小，通常可存 1000+ key
- 3 层 B 树可索引 10亿 数据，只需 3 次磁盘 I/O
```

### 3.2 B 树查找

```
查找流程:

查找 45:

          [30]
         /    \
      [20]    [40, 50]
      /  \    /   |   \
   [10][25][35] [45] [60,70]

Step 1: 从根开始，45 > 30 → 走右子树
Step 2: 节点 [40, 50]，45 在 40 和 50 之间
Step 3: 走中间子树，找到 [45]

查找次数: 3 次（树高）
对比二叉树: 相同数据，二叉树高度 ≈ log₂(10亿) ≈ 30
B 树优势: 树高低，减少磁盘 I/O
```

### 3.3 B 树插入

```
插入 38 到 B 树:

Before:
          [30]
         /    \
      [20]    [40, 50]
      /  \    /   |   \
   [10][25][35] [45] [60,70]

Step 1: 找到插入位置
        38 > 30 → 右子树
        38 在 40 和 50 之间 → 中间子树
        叶子节点 [35] 只有 1 个 key，可以插入

Step 2: 插入到叶子节点
          [30]
         /    \
      [20]    [40, 50]
      /  \    /   |   \
   [10][25][35,38] [45] [60,70]


插入 42（触发分裂）:

Before:
          [30]
         /    \
      [20]    [40, 50]
      /  \    /   |   \
   [10][25][35,38] [45] [60,70]

Step 1: 找到插入位置 → [40, 50] 的叶子节点 [45]
Step 2: 45 节点已满（2 个 key），需要分裂！

Step 3: 分裂节点 [45, 42]（假设已排序 [42, 45]）
        - 中间 key 42 提升到父节点
        - 左半部分 [ ] 作为左子节点（空，保留 42 的位置）
        - 右半部分 [45] 作为右子节点
        
        Wait，这个示例有问题。正确的 3 阶 B 树中，
        叶子节点最多 2 个 key。
        
        正确的分裂过程:
        
        插入 42 到 [45]:
        节点变成 [42, 45]，超过最大 2 个 key
        
        分裂:
        - 中间 key: 42（或 45，取中间）
        - 左节点: [ ]
        - 右节点: [45]
        
        提升 42 到父节点 [40, 50]:
        父节点变成 [40, 42, 50]，超过最大 2 个 key！
        
        继续分裂父节点:
        - 中间 key: 42
        - 左节点: [40]
        - 右节点: [50]
        
        提升 42 到根节点 [30]:
        根节点变成 [30, 42]，未超过 2 个 key ✓
        
After:
            [30, 42]
           /    |    \
        [20]  [40]  [50]
        /  \    |    /   \
     [10][25][ ] [45] [60,70]

树高增加！
```

### 3.4 B+ 树（B 树的变种）

```
B+ 树结构（MySQL InnoDB 使用）:

内部节点（只存 key，不存数据）:
            [30]
           /    \
        [20]    [40]
        /  \    /    \
      ...  ... ...   ...
      
叶子节点（存 key + 数据，形成链表）:
[10] → [20] → [25] → [30] → [35] → [40] → [45] → ...
  │      │       │      │       │      │       │
 data   data    data   data    data   data    data

B+ 树优势:
1. 内部节点更小，可容纳更多 key，树高更低
2. 叶子节点形成链表，范围查询更快
3. 所有数据在叶子节点，查询稳定
```

说明：

- 便于理解"节点内二分 + 子树下探"思想

## 5. 最佳实践

1. **数据库场景优先使用成熟存储引擎实现**。  
2. **自研时先从查询实现入手，再扩展插删**。  
3. **节点大小要结合页大小（如 4KB/16KB）调优**。  
4. **关键路径中尽量减少对象分配**。  
5. **通过随机与有序数据混合测试分裂/合并逻辑**。

### B 树 vs B+ 树

```
特性对比:

                    B树                 B+树
数据存储位置:     所有节点             仅叶子节点
叶子节点关系:      无关联               链表连接
内部节点作用:     存 key + 数据指针     仅存 key
范围查询:         中序遍历             叶子节点顺序扫描
查询稳定性:        O(log n) ~ O(log n)  稳定 O(log n)
空间利用率:        较低                 较高

实际应用:
- B 树: 文件系统（NTFS, HFS+）
- B+ 树: 数据库索引（MySQL, PostgreSQL, MongoDB）
```
