# Java性能优化实战

## 1. 性能优化方法论

```
性能优化三步法:

1. 度量（Measure）
   - 找出瓶颈（CPU/内存/IO/网络）
   - 使用 profiling 工具
   - 建立基准指标

2. 分析（Analyze）
   - 定位热点代码
   - 理解瓶颈根因
   - 评估优化收益

3. 优化（Optimize）
   - 针对性优化
   - 验证效果
   - 避免过度优化

黄金法则:
"过早优化是万恶之源" —— Donald Knuth
先写清晰正确的代码，再优化真正的瓶颈
```

## 2. JVM性能调优

### 2.1 内存模型与GC

```
JVM 内存结构:

┌─────────────────────────────┐
│        堆（Heap）            │
├─────────────┬───────────────┤
│  年轻代      │    老年代      │
│  Young      │     Old       │
│             │               │
│ ┌───┬───┐   │               │
│ │Eden│S0│S1│ │               │
│ └───┴───┘   │               │
│  8:1:1      │               │
└─────────────┴───────────────┘

GC 算法选择:

Serial:         单线程，小内存（< 100MB）
Parallel:       多线程，吞吐量优先（批处理）
CMS:            低延迟，已废弃（JDK14移除）
G1:             平衡吞吐量和延迟（JDK9+默认）
ZGC:            超低延迟（< 10ms），JDK11+
Shenandoah:     超低延迟，Red Hat开发

G1 GC 调优参数:
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200          # 目标最大停顿时间
-XX:G1HeapRegionSize=16m          # Region大小
-XX:InitiatingHeapOccupancyPercent=45  # 触发并发标记的阈值

GC 日志分析:
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:/path/to/gc.log
```

### 2.2 内存泄漏排查

```
内存泄漏症状:
- 老年代持续增长
- Full GC 频繁但内存不释放
- OOM: Java heap space

排查工具:
1. jmap 生成堆转储
   jmap -dump:format=b,file=heap.hprof <pid>

2. MAT（Memory Analyzer Tool）分析
   - Dominator Tree: 查看大对象
   - Leak Suspects: 自动泄漏分析
   - Path to GC Roots: 找到引用链

常见泄漏场景:
1. 静态集合持有对象引用
2. ThreadLocal 未 remove
3. 监听器未注销
4. 数据库连接未关闭
5. 缓存无限增长（无过期策略）
```

### 2.3 线程Dump分析

```bash
# 生成线程Dump
jstack -l <pid> > threads.txt

# 分析工具
- fastthread.io（在线分析）
- IBM Thread and Monitor Dump Analyzer

关键状态:
RUNNABLE:     正在运行
BLOCKED:      等待监视器锁（死锁 suspects）
WAITING:      无限期等待（wait/join）
TIMED_WAITING: 限时等待（sleep/wait(timeout)）
```

## 3. 代码层面优化

### 3.1 集合优化

```java
// 1. 预估容量，避免扩容
List<String> list = new ArrayList<>(10000);  // 已知大小

// 2. 选择合适的集合
// 读多写少: CopyOnWriteArrayList
// 高并发Map: ConcurrentHashMap
// 排序去重: TreeSet
// FIFO队列: ArrayDeque (比LinkedList快)

// 3. 避免在循环中创建对象
// 差:
for (String s : list) {
    String[] parts = s.split(",");  // 每次创建Pattern
}

// 好:
Pattern pattern = Pattern.compile(",");
for (String s : list) {
    String[] parts = pattern.split(s);
}

// 4. Stream 优化
// 差: 多次遍历
list.stream().filter(...).map(...).collect(...)
    .stream().filter(...).collect(...);

// 好: 链式操作，单次遍历
list.stream()
    .filter(...)
    .map(...)
    .filter(...)
    .collect(...);
```

### 3.2 字符串优化

```java
// 1. StringBuilder vs StringBuffer
// 单线程: StringBuilder（无锁，更快）
// 多线程: StringBuffer（同步）

// 2. 避免字符串拼接（循环中）
// 差:
String result = "";
for (String s : list) {
    result += s;  // 每次创建新String对象
}

// 好:
StringBuilder sb = new StringBuilder();
for (String s : list) {
    sb.append(s);
}
String result = sb.toString();

// 3. 使用intern()减少内存（谨慎）
String s1 = new String("hello").intern();
String s2 = new String("hello").intern();
// s1 == s2 为 true

// 4. Split优化
// JDK8+ String.split 使用快速路径（单字符、双字符）
// 复杂正则预编译 Pattern
```

### 3.3 I/O优化

```java
// 1. 使用Buffered包装
// 差:
FileInputStream fis = new FileInputStream("file.txt");

// 好:
BufferedInputStream bis = new BufferedInputStream(
    new FileInputStream("file.txt"));

// 2. NIO for 大文件
Path path = Paths.get("large-file.txt");
Files.lines(path).forEach(System.out::println);

// 3. 对象序列化优化
// 使用 Protocol Buffers / Kryo 替代 Java Serialization
// 避免 Serializable 的反射开销

// 4. 数据库连接池
// HikariCP: 性能最佳
HikariConfig config = new HikariConfig();
config.setMaximumPoolSize(10);      // 根据CPU核数设置
config.setMinimumIdle(5);
config.setConnectionTimeout(30000);
config.setIdleTimeout(600000);
config.setMaxLifetime(1800000);
```

## 4. 数据库优化

### 4.1 SQL优化

```sql
-- 1. 避免SELECT *
SELECT id, name FROM user WHERE id = 1;

-- 2. 索引优化
-- 单列索引: WHERE、JOIN、ORDER BY 列
-- 复合索引: 最左前缀原则 (a,b,c) 可支持 a、ab、abc
-- 覆盖索引: SELECT 列全在索引中，无需回表

-- 3. 避免函数操作索引列
-- 差: WHERE YEAR(create_time) = 2024
-- 好: WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01'

-- 4. 分页优化
-- 差: OFFSET 1000000（大量数据丢弃）
-- 好: 游标分页
SELECT * FROM user WHERE id > 1000000 ORDER BY id LIMIT 100;

-- 5. 批量操作
-- 单条: INSERT INTO user VALUES (...)
-- 批量: INSERT INTO user VALUES (...), (...), (...)...
-- JDBC batch: preparedStatement.addBatch()
```

### 4.2 连接池配置

```
HikariCP最佳实践:

maximumPoolSize = (core_count * 2) + effective_spindle_count
                = CPU核数 * 2 + 磁盘数
                
典型配置:
minimumIdle = maximumPoolSize  （固定连接数，避免创建开销）
connectionTimeout = 30000      （30秒等待）
idleTimeout = 600000           （10分钟空闲回收）
maxLifetime = 1800000          （30分钟强制重建，避免连接失效）

监控指标:
- activeConnections: 活跃连接数
- idleConnections: 空闲连接数
- pendingThreads: 等待连接的线程数
- totalConnections: 总连接数
```

## 5. 缓存优化

```
缓存策略:

1. 本地缓存（Caffeine）
   适用: 读多写少、数据量小、单机部署
   
   LoadingCache<String, User> cache = Caffeine.newBuilder()
       .maximumSize(1000)
       .expireAfterWrite(10, TimeUnit.MINUTES)
       .refreshAfterWrite(1, TimeUnit.MINUTES)
       .build(key -> loadFromDB(key));

2. 分布式缓存（Redis）
   适用: 多节点共享、数据量大
   
   缓存模式:
   - Cache-Aside: 应用管理缓存（最常用）
   - Read-Through: 缓存自动加载
   - Write-Through: 同步写缓存和DB
   - Write-Behind: 异步写DB

3. 多级缓存
   L1: Caffeine（本地）
   L2: Redis（分布式）
   L3: DB（持久化）
   
   读取顺序: L1 → L2 → L3
   写入顺序: DB → 删除 L2 → 删除 L1

缓存问题:
- 缓存穿透: 查询不存在数据 → 布隆过滤器/空值缓存
- 缓存击穿: 热点key过期 → 互斥锁/逻辑过期
- 缓存雪崩: 大量key同时过期 → 随机过期时间
```

## 6. 压测与监控

### 6.1 JMeter压测

```
JMeter关键指标:

吞吐量 (Throughput): 每秒处理请求数 (TPS/QPS)
响应时间 (Response Time):
  - 平均响应时间
  - P50/P90/P95/P99（百分位延迟）
错误率 (Error Rate): 失败请求占比
并发数 (Concurrency): 同时处理的请求数

压测步骤:
1. 基准测试（单用户）→ 确认逻辑正确
2. 容量测试（逐步加压）→ 找到性能拐点
3. 稳定性测试（持续运行）→ 检查内存泄漏
4. 峰值测试（瞬间高压）→ 验证限流降级

JMeter配置:
线程组: 1000 并发
Ramp-up: 60 秒（逐步加压）
持续时间: 300 秒（持续5分钟）
```

## 7. 性能优化 checklist

```
代码层:
□ 避免创建不必要的对象
□ 使用StringBuilder替代字符串拼接
□ 选择合适的集合初始容量
□ 懒加载（Lazy Initialization）
□ 缓存重复计算结果
□ 使用基本类型替代包装类型
□ 避免在循环中调用耗时操作

JVM层:
□ 选择合适的GC算法
□ 调整堆内存大小（-Xms = -Xmx）
□ 开启GC日志分析
□ 排查内存泄漏
□ 使用JMH进行微基准测试

数据库层:
□ 添加合适的索引
□ 优化慢SQL（EXPLAIN分析）
□ 使用连接池（HikariCP）
□ 批量操作替代单条
□ 读写分离
□ 分库分表（数据量大时）

架构层:
□ 使用缓存（本地+分布式）
□ 异步处理（MQ）
□ 服务降级熔断
□ 限流防刷
□ CDN加速静态资源
□ 数据库主从复制
```
