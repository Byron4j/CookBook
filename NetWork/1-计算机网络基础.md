# 计算机网络基础

## 1. OSI 七层模型与 TCP/IP 四层模型

```
OSI 七层模型              TCP/IP 四层模型

7. 应用层                  
6. 表示层         ────┐    应用层（HTTP/FTP/DNS）
5. 会话层         ────┘
4. 传输层                  传输层（TCP/UDP）
3. 网络层                  网络层（IP/ICMP）
2. 数据链路层              网络接口层（以太网/WiFi）
1. 物理层

各层职责:
应用层: 为用户应用程序提供网络服务
传输层: 端到端通信、流量控制、差错控制
网络层: 寻址、路由选择、分组转发
网络接口层: 物理传输、帧封装、MAC寻址

数据封装过程:

应用数据
  ↓ + 应用层头部
应用层 PDU
  ↓ + 传输层头部 (TCP/UDP)
段/报文 (Segment)
  ↓ + 网络层头部 (IP)
分组/包 (Packet)
  ↓ + 数据链路层头部 (MAC)
帧 (Frame)
  ↓
比特流 (Bits)
```

## 2. TCP 与 UDP

### 2.1 核心区别

| 特性 | TCP | UDP |
|------|-----|-----|
| 连接性 | 面向连接 | 无连接 |
| 可靠性 | 可靠传输（确认、重传） | 不可靠（尽力而为） |
| 顺序性 | 保证数据顺序 | 不保证顺序 |
| 流量控制 | 有（滑动窗口） | 无 |
| 拥塞控制 | 有 | 无 |
| 头部开销 | 20 字节 | 8 字节 |
| 传输效率 | 较低 | 较高 |
| 适用场景 | HTTP、文件传输、邮件 | DNS、视频流、游戏 |

### 2.2 TCP 三次握手

```
三次握手建立连接:

客户端                              服务端
   │ ───── SYN, seq=x ────────>      │
   │        （请求建立连接，序号 x）     │
   │                                  │
   │ <─── SYN+ACK, seq=y, ack=x+1 ─── │
   │        （同意建立，序号 y，期望 x+1）│
   │                                  │
   │ ───── ACK, ack=y+1 ──────>      │
   │        （确认收到，期望 y+1）       │
   │                                  │
   │        连接建立，可以传输数据        │

为什么是三次？
- 两次：客户端无法确认服务端是否收到自己的确认
- 四次：多余，合并 SYN+ACK 更高效
```

### 2.3 TCP 四次挥手

```
四次挥手断开连接:

客户端                              服务端
   │ ───── FIN, seq=u ────────>      │
   │        （请求断开）                │
   │                                  │
   │ <──── ACK, ack=u+1 ──────       │
   │        （确认收到断开请求）         │
   │        （此时服务端可能还有数据发送） │
   │                                  │
   │ <──── FIN, seq=w ────────       │
   │        （服务端也请求断开）         │
   │                                  │
   │ ───── ACK, ack=w+1 ──────>      │
   │        （确认收到）                │
   │                                  │
   │        等待 2MSL 后完全关闭       │

TIME_WAIT 状态（2MSL）:
- 确保最后的 ACK 能被服务端收到
- 防止已失效的连接请求报文出现在本连接中
- MSL = Maximum Segment Lifetime（报文最大生存时间）
```

### 2.4 TCP 滑动窗口

```
滑动窗口机制:

发送端窗口:
已发送并确认 │ 已发送未确认 │ 允许发送但未发送 │ 不允许发送
    ▲            ▲            ▲
    │            │            │
    └────────────┴────────────┘
         发送窗口（大小由接收端决定）

流量控制:
接收端通过 Window Size 字段告知发送端当前可接收的数据量

拥塞控制（四种算法）:
1. 慢启动（Slow Start）
   拥塞窗口 cwnd 从 1 开始，每轮翻倍
   
2. 拥塞避免（Congestion Avoidance）
   cwnd 超过阈值后，每轮线性增长（+1）
   
3. 快重传（Fast Retransmit）
   收到 3 个重复 ACK，立即重传，不等超时
   
4. 快恢复（Fast Recovery）
   快重传后，不执行慢启动，而是调整阈值后继续拥塞避免
```

## 3. HTTP/HTTPS

### 3.1 HTTP 演进

```
HTTP/1.0 (1996):
- 短连接：每个请求/响应后关闭 TCP
- 队头阻塞（Head-of-Line Blocking）

HTTP/1.1 (1997):
- 持久连接（Keep-Alive）
- 管道化（Pipelining）：允许多个请求，但响应必须按序
- 分块传输编码（Chunked Transfer Encoding）
- 缓存控制（Cache-Control）

HTTP/2 (2015):
- 二进制分帧（Binary Framing）
- 多路复用（Multiplexing）：单一 TCP 连接上并行传输
- 头部压缩（HPACK）
- 服务器推送（Server Push）

HTTP/3 (2022):
- 基于 QUIC（UDP 之上）
- 彻底解决队头阻塞
- 连接迁移（切换网络不影响连接）
- 更快的握手（0-RTT）
```

### 3.2 HTTP 状态码

```
1xx: 信息
   100 Continue - 继续发送请求体
   
2xx: 成功
   200 OK
   201 Created - 资源创建成功
   204 No Content - 无返回内容
   206 Partial Content - 范围请求成功
   
3xx: 重定向
   301 Moved Permanently - 永久重定向
   302 Found - 临时重定向
   304 Not Modified - 缓存有效
   
4xx: 客户端错误
   400 Bad Request - 请求格式错误
   401 Unauthorized - 未认证
   403 Forbidden - 无权限
   404 Not Found - 资源不存在
   429 Too Many Requests - 限流
   
5xx: 服务端错误
   500 Internal Server Error
   502 Bad Gateway - 网关错误
   503 Service Unavailable - 服务不可用
   504 Gateway Timeout - 网关超时
```

### 3.3 HTTPS 握手过程

```
TLS/SSL 握手（RSA 密钥交换）:

客户端                              服务端
   │ ───── Client Hello ─────>      │
   │  支持的 TLS 版本、加密套件、     │
   │  客户端随机数                   │
   │                                  │
   │ <──── Server Hello ─────       │
   │  选定的 TLS 版本、加密套件、     │
   │  服务端随机数、证书              │
   │                                  │
   │  验证证书（CA 公钥验签）          │
   │  提取服务端公钥                   │
   │                                  │
   │ ───── Client Key Exchange ─>   │
   │  生成 Pre-master Secret          │
   │  用服务端公钥加密后发送           │
   │                                  │
   │  双方用 Client Random +          │
   │  Server Random + Pre-master      │
   │  生成 Session Keys               │
   │                                  │
   │ ───── Change Cipher Spec ─>    │
   │  后续通信使用对称加密             │
   │ <──── Change Cipher Spec ────   │
   │                                  │
   │        加密通信开始               │

TLS 1.3 优化:
- 握手仅需 1-RTT（甚至 0-RTT）
- 删除过时算法，只保留 AEAD 加密
- Server Hello 后所有消息加密
```

## 4. DNS 解析

```
DNS 解析流程:

用户输入 www.example.com
        │
        ▼
┌───────────────┐
│ 浏览器缓存     │ ← 检查是否近期访问过
└───────┬───────┘
        │ 未命中
        ▼
┌───────────────┐
│ OS 缓存        │ ← hosts 文件、系统 DNS 缓存
└───────┬───────┘
        │ 未命中
        ▼
┌───────────────┐
│ 本地 DNS       │ ← 路由器/ISP 提供的 DNS
│ (递归解析)     │
└───────┬───────┘
        │ 未命中
        ▼
┌───────────────┐     ┌───────────────┐
│ 根 DNS 服务器  │ ──→ │ .com 顶级域    │
│ (Root)        │     │ (TLD)         │
└───────────────┘     └───────┬───────┘
                              │
                              ▼
                        ┌───────────────┐
                        │ example.com   │
                        │ 权威 DNS       │
                        └───────┬───────┘
                                │
                                ▼
                        返回 IP: 93.184.216.34

DNS 记录类型:
A     → IPv4 地址
AAAA  → IPv6 地址
CNAME → 别名记录
MX    → 邮件交换记录
TXT   → 文本记录（SPF、DKIM）
NS    → 域名服务器记录
```

## 5. WebSocket

```
WebSocket 全双工通信:

HTTP 握手升级:

客户端请求:
GET /chat HTTP/1.1
Host: server.example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13

服务端响应:
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

之后:
TCP 连接保持，双方可随时发送数据帧

WebSocket vs HTTP 轮询:

轮询:
客户端 ──→ 有消息吗？ ──→ 服务端
  ↑←──── 没有 ───────────┘
  │（等待 5 秒）
  ├──→ 有消息吗？ ──→ 服务端
  ↑←──── 没有 ───────────┘
  （大量无效请求）

WebSocket:
客户端 ←──────── 连接保持 ───────→ 服务端
  │                                  │
  ├──→ 消息 ────────────────────────→│
  │←─────────────────────────────────┤←── 消息
  │                                  │
  （实时双向，开销小）
```

## 6. CDN 内容分发网络

```
CDN 工作原理:

用户访问 www.example.com/image.jpg
        │
        ▼
┌───────────────┐
│  DNS 解析      │ ← CNAME 指向 CDN 域名
│  返回最近节点  │
└───────┬───────┘
        │
        ▼
┌───────────────┐
│  CDN 边缘节点  │ ← 检查缓存
│  北京节点      │
└───────┬───────┘
        │
   命中缓存？
   ├── 是 → 直接返回（最快）
   │
   └── 否 → 回源到源站获取
            同时缓存到边缘节点

CDN 优势:
- 减少源站压力
- 降低网络延迟（就近访问）
- 带宽优化
- 抗 DDoS 攻击

缓存策略:
Cache-Control: max-age=3600  （缓存 1 小时）
Cache-Control: no-cache       （需重新验证）
Cache-Control: no-store       （不缓存）
ETag + If-None-Match          （协商缓存）
```

## 7. 网络安全基础

```
常见攻击与防护:

SQL 注入:
攻击: username = "admin' OR '1'='1"
防护: 使用预编译语句（PreparedStatement）
      参数化查询

XSS（跨站脚本）:
攻击: <script>stealCookie()</script>
防护: 输入过滤、输出编码（HTML Entity）
      CSP（Content Security Policy）

CSRF（跨站请求伪造）:
攻击: 诱导用户点击恶意链接，以用户身份执行操作
防护: CSRF Token、SameSite Cookie、Referer 检查

中间人攻击:
攻击: 拦截并篡改通信数据
防护: HTTPS（TLS 加密）、证书固定（Pinning）

DDoS:
攻击: 大量请求耗尽服务器资源
防护: CDN、WAF、限流、扩容
```

## 8. 最佳实践

1. **使用 HTTPS**：所有生产环境强制 TLS 1.2+
2. **启用 HTTP/2**：提升传输效率
3. **合理配置 Keep-Alive**：减少 TCP 握手开销
4. **使用 CDN**：静态资源加速，减轻源站压力
5. **DNS 预解析**：`<link rel="dns-prefetch" href="//cdn.example.com">`
6. **连接复用**：连接池管理，避免频繁创建连接
7. **超时配置**：连接超时、读取超时、写入超时合理设置
8. **重试策略**：网络抖动时优雅重试，避免雪崩
